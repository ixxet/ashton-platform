# Tracer Closure-Hardening Template

Use this template when a tracer already works, but still needs a deliberate
hardening pass before it can be called closure-clean.

This is not feature expansion. It is the pass that turns "behavior exists" into
"the slice is trustworthy, auditable, and ready to tag."

## Purpose

Use closure hardening when any of the following are still true after the first
implementation pass:

- the producer and consumer can still drift on the active wire contract
- runtime validation is weaker than the contract claims
- smoke coverage is missing on the real execution path
- logs are too weak to debug rejects or duplicates
- repo-local docs and control-plane docs do not yet match reality
- the tracer works, but the closure story is still soft

## Standard

Closure hardening should over-engineer:

- correctness
- contract fidelity
- negative tests
- auditability
- release discipline

Closure hardening should not widen:

- feature scope
- product breadth
- speculative abstractions
- adjacent systems that are not required to close the tracer

## Working Loop

Every slice should follow the same loop:

1. change one thing
2. run only the minimum relevant tests immediately
3. record any real mistake or revelation in the repo that learned it
4. commit the slice with a narrow message
5. push the slice before moving on
6. do not continue while red

## Scope Table

Fill this before writing code.

| Status | Item | Why It Is In Or Out | Owner Repo |
| --- | --- | --- | --- |
| In scope | active contract consumption or enforcement | producer and consumer drift must close before tagging | affected repos |
| In scope | semantic validation | malformed or semantically invalid events must not mutate state | affected repos |
| In scope | negative tests and smoke tests | closure requires proof, not only code changes | affected repos |
| In scope | docs, runbooks, tracer matrix, release notes | control-plane gate must match reality | repo-local docs, then `ashton-platform` |
| Out of scope | adjacent product slices | do not widen beyond the tracer boundary | later tracer |

Replace the generic rows above with tracer-specific entries before starting.

## Closure Gates

The tracer is not closed until all four gates pass:

### 1. Behavioral Gate

- the claimed end-to-end slice works
- duplicate and replay behavior are deterministic
- the tracer does not mutate adjacent state that it does not own

### 2. Contract Gate

- one runtime contract source of truth exists
- producer and consumer cannot silently drift on field names or allowed values
- tests prove that shared bytes are accepted unchanged

### 3. Operational Gate

- focused tests pass during each slice
- full repo test matrix passes before closure
- at least one real smoke path runs on the live protocol surface
- failure signaling is readable in logs

### 4. Control-Plane Gate

- repo-local docs match reality
- `growing-pains` records the real findings
- `ashton-platform` is updated only after code and tests are green
- tags are cut only after the control-plane docs are current

## Contract Hardening Checklist

Use this when the tracer crosses repos or services.

- identify the active message, API, schema, or storage contract
- remove duplicate local definitions if a shared contract repo already exists
- centralize subject names, enum values, and timestamp expectations
- decide whether runtime code imports shared types directly, uses a shared
  helper, or uses a shared validator
- prove the consumer accepts producer-generated bytes without hand-editing
- reject semantically invalid values, not just malformed JSON

## Validation Checklist

Reject or handle explicitly:

- wrong service source
- wrong event or route type
- unsupported enum values
- missing required identifiers
- missing or invalid timestamps
- anonymous payloads if the tracer says they are ignored
- duplicates
- open-state collisions
- unknown identity mappings
- broker or dependency failures

## Core Smoke Test Template

Use a real process path whenever practical, not only stubs.

| Case | Setup | Command Or Action | Expected Result |
| --- | --- | --- | --- |
| Health surface | start service(s) with tracer config | hit health endpoint or health command | service reports healthy and tracer dependency state |
| Happy path | publish one valid input | run the real producer path | one downstream state mutation occurs |
| Readback path | query downstream state | use the real read surface | exactly one expected record is visible |
| Replay | repeat the same input | rerun producer or republish bytes | duplicate outcome is explicit and state does not multiply |
| Worker path | enable the background worker path | run the periodic or queue consumer mode | the same end-to-end result occurs without the one-shot helper |

## Edge-Case Smoke Test Template

Do not close a tracer with only the happy path.

| Case | Input Or Trigger | Expected Outcome | Classification |
| --- | --- | --- | --- |
| Missing required field | omit required identifier or timestamp | reject clearly, no state change | negative |
| Invalid timestamp | malformed timestamp string | reject clearly, no state change | negative |
| Unsupported enum | send unknown source or mode | reject clearly, no state change | negative |
| Anonymous event | send empty identity where tracer says ignore | explicit ignore, no state change | defensive |
| Unknown identity | send unmapped identifier | explicit unknown outcome, no state change | business rule |
| Duplicate event id | replay same source event | duplicate outcome, no second record | idempotency |
| New event while state is already open | send second open arrival | already-open outcome, original state preserved | idempotency |
| Dependency outage | stop broker or backing store | failure is logged clearly and does not corrupt state | resiliency |
| Restart or replay | restart consumer and reprocess retained input if relevant | deterministic result under replay rule | resiliency |

Add tracer-specific rows when the domain has more sharp edges.

## Repo-By-Repo Packet Template

For each repo in scope, define:

- goal of the repo in this hardening pass
- exact files expected to change
- exact tests to run after each slice
- exact commit messages to use
- docs that must be updated if reality changed

Example packet shape:

### Repo: `<repo-name>`

- Purpose:
  what this repo must harden and what it must not widen into
- Work packet A:
  smallest code or contract change
- Tests:
  minimum test command set for packet A
- Commit:
  exact narrow commit message
- Docs:
  repo-local docs that must change if packet A changes operator reality

## Findings Template

Every real issue discovered during hardening should be recorded in the repo that
learned it and summarized in the control-plane closure note.

Use this shape:

- Symptom: what failed, drifted, or stayed too soft
- Cause: why it happened
- Fix: what changed
- Rule: what future tracers must do instead

## Deferrals Table Template

A tracer should distinguish between:

- fixed in this pass
- intentionally deferred
- still blocked and not fixed

Use one table so the closeout is auditable.

| Type | Item | Status | Why It Was Not Done Here | Future Owner |
| --- | --- | --- | --- | --- |
| Feature | departure handling | deferred | tracer only proves arrival -> visit creation | later visit-closing tracer |
| Feature | workout creation | deferred | visit history is separate from workouts | later workout tracer |
| Bug | known non-blocking defect | unresolved | blocked by external dependency or wider refactor | named follow-up |

If there are no unresolved bugs, say so explicitly above the table.

## Documentation Housekeeping

Before tagging, update only the docs that reflect changed reality:

- repo `README.md`
- repo `docs/roadmap.md`
- repo `docs/growing-pains.md`
- repo runbook only if the operating procedure changed
- `ashton-platform` tracer matrix
- `ashton-platform` implementation board if the next blocker sequence changed

Do not rewrite long architecture essays unless the change is broad and stable.

## Commit Discipline

Use small narrative commits. Good patterns:

- `refactor: consume shared <contract> in <repo>`
- `fix: enforce <semantic rule>`
- `test: add <edge case> coverage`
- `docs: align <tracer> closure docs`

Prefer one logical idea per commit. Push each verified slice so GitHub history
shows the hardening narrative, not just the final state.

## Release Sequence

1. finish repo-local hardening and push all slice commits
2. rerun the full test matrix for the touched repos
3. complete manual smoke where required
4. update repo-local docs
5. update `ashton-platform`
6. merge to `main`
7. tag from `main`
8. delete the working branch if it is no longer needed

## Final Report Template

The closure report should include:

- what changed
- exact commits by repo
- exact automated commands run
- exact manual smoke path run
- findings discovered during hardening
- deferrals table
- whether any unresolved bugs remain
- tag readiness or final tags
- branch cleanup status
